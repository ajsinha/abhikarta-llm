Overview

The Abhikarta LLM Chat Interface is a powerful, multi-provider conversational AI platform that allows you to interact with various Large Language Models (LLMs) through a unified interface. Whether you're using OpenAI's GPT, Anthropic's Claude, Google's Gemini, or other providers, our platform provides seamless access to cutting-edge AI capabilities.

Multi-Provider Support

Access 13+ AI providers including OpenAI, Anthropic, Google, Cohere, Mistral, Meta, Groq, HuggingFace, and more.

Conversation History

Full context retention throughout your conversation, enabling coherent multi-turn dialogues.

Parameter Control

Fine-tune model behavior with temperature, max tokens, and top-p sampling controls.

Session Caching

Optimized performance with intelligent facade caching for faster responses.

Interaction Logging

All interactions are logged to the database for auditing and analysis.

Registry Reload

Dynamically reload provider registry without restarting the application.

Getting Started

Follow these simple steps to start your first conversation with an AI model:

  1. Navigate to Chat

    Access the chat interface by clicking on "Chat" in the main navigation menu, or visit /llm/chat directly.

  2. Select a Provider

    Choose from the available AI providers in the "Provider" dropdown. Each provider offers different models with unique capabilities.

    Tip: Start with OpenAI or Anthropic if you're new to AI chat interfaces.
  3. Select a Model

    After selecting a provider, choose a specific model from the "Model" dropdown. Each model has different capabilities, context windows, and pricing.

    Tip: Check the model information that appears below the dropdown to see context window size and capabilities.
  4. Adjust Parameters (Optional)

    Fine-tune the model's behavior using the parameter sliders. Default values work well for most use cases.

  5. Start Chatting

    Type your message in the input box and click the send button or press Enter to send your message.

  6. Review Response

    The AI model will process your request and display the response in the chat window. You can continue the conversation by sending more messages.

Success! You're now ready to have intelligent conversations with advanced AI models!

Interface Layout

The chat interface is organized into several key areas:

1. Chat Header

Located at the top, displaying the application name and current session ID. The session ID is used for tracking and caching purposes.

Session ID: chat_a1b2c3d4_1732723200

2. Configuration Panel

The configuration panel contains all controls for selecting and configuring your AI interaction:

Status Indicator

Shows the current system status:

  • Loading... - System is initializing or loading data
  • Ready - System is ready for interaction
  • Select provider and model - Waiting for configuration
  • Error - System encountered an error

Provider Selection

Dropdown menu showing available AI providers. The number in parentheses indicates how many models are available from that provider.

Model Selection

After selecting a provider, this dropdown populates with available models. Each model shows its display name.

Model Information

When a model is selected, detailed information appears:

  • Context Window: Maximum number of tokens the model can process (input + output)
  • Max Output Tokens: Maximum tokens the model can generate in a single response
  • Description: Brief description of the model's capabilities

Action Buttons

  • Reload Registry: Refresh the provider and model registry without restarting
  • New Chat: Start a new conversation (clears history and creates new session)
  • End Chat: End current session and clean up resources

3. Chat Messages Panel

The main conversation area where messages appear:

Message Types

  • User Messages: Your messages appear on the right in purple gradient bubbles
  • Assistant Messages: AI responses appear on the left in white bubbles with gray borders
  • Typing Indicator: Animated dots show when the AI is processing your request

Message Metadata

Each message displays:

  • Role: "You" or "Assistant"
  • Content: The actual message text
  • Timestamp: When the message was sent/received
  • Cache Indicator: Small dot showing if cached facade was used (for performance)

4. Chat Input Area

Located at the bottom of the chat panel:

  • Text Input: Type your message here (supports multi-line input)
  • Send Button: Click to send or press Enter
  • Keyboard Shortcuts:
    • Enter - Send message
    • Shift + Enter - New line in message

5. Parameters Panel

Fine-tune model behavior with these controls:

  • Temperature: Slider controlling randomness (0.0 - 2.0)
  • Max Tokens: Maximum response length (100 - 32000)
  • Top P: Nucleus sampling parameter (0.0 - 1.0)

Model Parameters Explained

Understanding and properly configuring these parameters can significantly improve your AI interactions:

Temperature (0.0 - 2.0)

Controls the randomness and creativity of the model's responses.

Value Behavior Use Cases
0.0 - 0.3 Very deterministic and focused Factual Q&A, code generation, data extraction
0.4 - 0.7 Balanced, default setting General conversation, analysis, explanations
0.8 - 1.2 More creative and varied Creative writing, brainstorming, storytelling
1.3 - 2.0 Highly random and experimental Extreme creativity, experimental outputs

Example: Temperature Effects

Prompt: "Write a tagline for a coffee shop"

  • Temperature 0.2: "Fresh coffee, great taste every day"
  • Temperature 0.7: "Where every cup tells a story"
  • Temperature 1.5: "Brewing liquid poetry in porcelain dreams"

Max Tokens (100 - 32000)

Controls the maximum length of the model's response.

  • Short Responses (100-500): Quick answers, summaries, simple questions
  • Medium Responses (500-2000): Detailed explanations, analysis, conversations
  • Long Responses (2000-8000): Articles, extensive analysis, code generation
  • Very Long (8000+): Long-form content, detailed documents, complex code
Important: Higher max tokens increase API costs and processing time. Set appropriately for your needs.

Top P (0.0 - 1.0)

Nucleus sampling parameter - controls diversity by limiting token selection to a cumulative probability.

  • 0.1 - 0.5: More focused, conservative responses
  • 0.6 - 0.9: Balanced diversity (default: 0.9)
  • 0.95 - 1.0: Maximum diversity, includes rare words
Best Practice: Use either temperature OR top_p, not both aggressively. If using high temperature, keep top_p at default.

Advanced Features

1. Session-Based Facade Caching

The platform implements intelligent caching to improve performance:

  • Automatic Caching: Model facades are cached per session and provider/model combination
  • Performance Boost: Reduces initialization overhead by ~90% for subsequent requests
  • TTL Management: Caches expire after 60 minutes of inactivity
  • Automatic Cleanup: Resources are cleaned up when chat sessions end
Tip: You'll see a small cache indicator dot next to assistant messages that used cached facades.

2. Conversation History Management

The system maintains full conversation context:

  • Context Retention: All previous messages in the session are sent to the model
  • Multi-Turn Dialogue: The model remembers earlier parts of the conversation
  • Session Isolation: Each new chat session starts fresh
  • History Clearing: Use "New Chat" to start a conversation from scratch

3. Interaction Logging

All interactions are logged to the database for:

  • Auditing: Track who used which models and when
  • Analysis: Understand usage patterns and model performance
  • Debugging: Investigate issues with specific interactions
  • Cost Tracking: Monitor token usage and API costs
Tip: Contact your administrator to access interaction logs and usage analytics.

4. Registry Reload

Dynamically update provider configurations without downtime:

  • Click "Reload Registry" to refresh provider and model lists
  • New providers added to configuration files become immediately available
  • Useful when administrators add new models or providers
  • No server restart required

5. Real-Time Streaming (Upcoming)

Future versions will support real-time streaming responses:

  • Token-by-token response streaming
  • Lower perceived latency for long responses
  • Ability to stop generation mid-stream
  • Progressive rendering of responses

Session Management

Understanding Chat Sessions

Each chat session has a unique identifier displayed in the header:

Session ID: chat_a1b2c3d4e5f6_1732723200 Format: chat_[random]_[timestamp]

Session Lifecycle

  1. Session Creation

    A new session is automatically created when you first visit the chat page or click "New Chat".

  2. Active Session

    During an active session, all messages and context are maintained. Model facades are cached for performance.

  3. Session End

    Clicking "End Chat" terminates the session and cleans up resources. Conversation history is cleared.

  4. New Session

    Clicking "New Chat" ends the current session and creates a fresh one with no prior context.

Best Practices for Session Management

Start New Chat When:
  • Switching to a completely different topic
  • The conversation has become confusing or off-track
  • You want to test different models on the same prompt
  • The context window is getting full (very long conversations)
Note: Ending a chat session clears all conversation history. Make sure to save any important information before ending.

Best Practices

Prompt Engineering Tips

Be Specific

Provide clear, detailed instructions. "Write a Python function to sort a list" is better than "write code".

Break It Down

For complex tasks, break them into multiple messages. Let the AI complete each step before proceeding.

Use Examples

Provide examples of desired input/output format. "Like this: [example]" helps the model understand your needs.

Set Context

Tell the model what role to play: "Act as a Python expert" or "Explain like I'm a beginner".

Iterate

If the response isn't quite right, ask for modifications: "Make it shorter" or "Add more detail".

Specify Constraints

Set clear boundaries: "In under 200 words" or "Without using external libraries".

Troubleshooting

Common Issues and Solutions

1. "Select provider and model" - Can't Send Messages

Cause: No provider or model selected

Solution:

  • Select a provider from the "Provider" dropdown
  • Wait for models to load
  • Select a model from the "Model" dropdown
  • Wait for "Ready" status before sending messages

2. Models Not Loading

Cause: Registry issue or provider configuration problem

Solution:

  • Click "Reload Registry" button
  • Wait a few seconds for the registry to reload
  • If problem persists, contact your administrator
  • Check browser console (F12) for error messages

3. "Error: Request failed" Messages

Possible Causes:

  • Network connectivity issues
  • API rate limits reached
  • Provider service outage
  • Invalid API credentials

Solutions:

  • Check your internet connection
  • Wait a moment and try again
  • Try a different provider or model
  • Contact administrator if issue persists

4. Responses Are Cut Off

Cause: Max tokens setting too low

Solution:

  • Increase the "Max Tokens" parameter
  • Typical values: 1000 for short, 2000 for medium, 4000+ for long responses
  • Check the model's maximum output limit

5. Model Gives Irrelevant Responses

Possible Causes:

  • Conversation context is too long or confusing
  • Temperature set too high
  • Prompt is ambiguous

Solutions:

  • Click "New Chat" to start fresh
  • Lower temperature to 0.3-0.5 for more focused responses
  • Rephrase your question more specifically
  • Break complex requests into simpler steps

6. Session Keeps Resetting

Cause: Browser session or cookie issues

Solution:

  • Check if cookies are enabled in your browser
  • Don't use incognito/private browsing mode
  • Clear browser cache and refresh
  • Make sure you're not logged out automatically

7. Slow Response Times

Possible Causes:

  • Provider API is experiencing high load
  • Very large context or max tokens setting
  • Network latency

Solutions:

  • Try a different provider (e.g., Groq for speed)
  • Reduce max_tokens if appropriate
  • Use "New Chat" to reduce context size
  • Try during off-peak hours

API Reference

For developers integrating with the chat system, here are the available API endpoints:

Chat Endpoints

GET /llm/chat

Renders the chat interface.

Authentication: Required (login)

Returns: HTML page with chat interface

POST /api/llm/chat/send

Send a message and get AI response.

Authentication: Required (login)

Request Body:

{ "provider": "openai", "model": "gpt-4", "message": "Your message here", "conversation_history": [ {"role": "user", "content": "Previous message"}, {"role": "assistant", "content": "Previous response"} ], "parameters": { "temperature": 0.7, "max_tokens": 1000, "top_p": 0.9 } }

Response:

{ "success": true, "response": "AI response text", "metadata": { "cached_facade": true, "provider": "openai", "model": "gpt-4", "timestamp": "2024-11-27T10:30:00" } }

POST /api/llm/chat/new

Create a new chat session.

Authentication: Required (login)

Response:

{ "success": true, "chat_session_id": "chat_a1b2c3d4_1732723200", "message": "New chat session created" }

POST /api/llm/chat/end

End current chat session and cleanup resources.

Authentication: Required (login)

Response:

{ "success": true, "message": "Chat session ended successfully" }

GET /api/llm/chat/session/info

Get current session information.

Authentication: Required (login)

Response:

{ "success": true, "session_id": "chat_a1b2c3d4_1732723200", "created_at": "2024-11-27T10:00:00", "message_count": 10, "active": true }

Registry Endpoints

GET /api/llm/providers

Get list of available providers.

Response:

{ "success": true, "providers": [ { "name": "openai", "display_name": "OpenAI", "model_count": 5 } ] }

GET /api/llm/models/{provider}

Get models for a specific provider.

Response:

{ "success": true, "models": [ { "name": "gpt-4", "display_name": "GPT-4", "context_window": 8192, "max_output": 4096, "description": "Most capable model" } ] }

POST /api/llm/registry/reload

Reload the provider registry.

Authentication: Required (login)

Response:

{ "success": true, "provider_count": 13, "message": "Registry reloaded successfully" }
