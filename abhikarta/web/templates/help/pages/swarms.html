Agent Swarms

LLM-powered choreography with event-driven multi-agent coordination

v1.3.0 Feature

What are Agent Swarms?

Agent Swarms enable multiple AI agents to work together on complex tasks through intelligent event-driven choreography. A Master Actor uses an LLM to coordinate agent activities, making dynamic decisions based on context and results.

Unlike traditional orchestration where workflows are predefined, swarms use choreography - the Master Actor observes and reacts, publishing events that agents subscribe to. This creates flexible, adaptive multi-agent systems.

Key Benefits
  • Intelligent task distribution
  • Dynamic agent scaling
  • Event-driven decoupling
  • Round-robin load balancing
  • External trigger support

Key Components

Master Actor

The intelligent choreographer that stays resident in memory. Uses an LLM to analyze incoming triggers, decide what tasks to publish, and aggregate results from agents.

Event Bus

Internal pub/sub system that decouples agents from the master. Supports topic patterns with wildcards (task.*, result.#), priority queuing, and message history.

Agent Pools

Pre-warmed pools of agent instances with configurable min/max sizes. Round-robin selection ensures fair distribution. Auto-scaling based on demand.

External Triggers

Swarms can be triggered by Kafka topics, RabbitMQ queues, HTTP webhooks, cron schedules, or direct user queries. Multiple triggers per swarm.

Round-Robin Selection

Database-backed instance tracking ensures fair load distribution. Selects idle instances by least-recently-used and use-count criteria.

Real-time Monitoring

Watch events flow through your swarm in real-time. Track execution history, agent metrics, and pool status. Debug with correlation IDs.

Round-Robin Agent Pool Selection

When the Master Actor needs to dispatch a task, it uses round-robin selection to pick an agent instance from the pool:

SELECT * FROM swarm_agent_instances WHERE swarm_id = ? AND agent_id = ? AND status = 'idle' ORDER BY COALESCE(last_used, '1970-01-01') ASC, use_count ASC LIMIT 1
Benefits
  • Zero startup latency
  • Fair load distribution
  • Prevents instance overload
  • Automatic failover
Pool Config
  • min_instances: Always keep warm
  • max_instances: Scale limit
  • auto_scale: Enable scaling
  • idle_timeout: Reclaim idle
Tracking
  • status: idle/busy
  • last_used: Timestamp
  • use_count: Total uses
  • tasks_completed: Success count
